## GATE RESULT: does NOT reproduce the shipped golden byte-for-byte

Command run (mirrors the README's documented example invocation — no `-c`, no `--preset`; the task
brief's assumed `--preset kir` does not exist in `run-t1k`, which only recognizes `hla[-wgs]`,
`kir-wgs`, `kir-wes`, and dies with "Unknown preset parameter kir" if passed a bare `kir`):

```
perl <t1k>/run-t1k -1 fixtures/example/example_1.fq -2 fixtures/example/example_2.fq \
  -f fixtures/example/ref/kir_rna_seq.fa -o example --od /tmp/kir_oracle -t 8
diff /tmp/kir_oracle/example_genotype.tsv fixtures/example/T1K_example_genotype.tsv
```

`diff` is **non-empty**. Investigation performed (see task report for full detail):

1. **Determinism ruled out**: repeated runs and `-t 1` vs `-t 8` are byte-identical to each other —
   the genotyper is confirmed deterministic regardless of thread count, matching spec.
2. **Code version ruled out**: in a read-only local clone of upstream T1K with full git history
   (not used as a vendor source per repo convention), `git
   log -- example/T1K_example_genotype.tsv` shows the golden file was last touched at commit
   `9376b55` — exactly the commit vendored into this repo. The Genotyper.cpp algorithm is therefore
   confirmed identical to whatever last regenerated the shipped golden.
3. **Reference gene-block order**: the golden's per-locus row order is alphabetical
   (`KIR2DL1..KIR2DL5B, KIR2DP1, KIR2DS1..KIR2DS5, KIR3DL1..KIR3DL3, KIR3DP1, KIR3DS1`); T1K assigns
   each gene an index in first-encounter order while scanning the `-f` reference FASTA
   (`Genotyper.hpp` `InitAlleleInfo()`), so gene row order in the output is a direct readout of allele
   order in the reference file. Both the fetched tarball's `kir_rna_seq.fa` and a fresh rebuild via
   `t1k-build.pl -d KIR.dat` (using the exact same bundled `KIR.dat`) place the two pseudogenes
   (`KIR2DP1`, `KIR3DP1`) at the very end, not interleaved alphabetically — i.e. the true encounter
   order in the only available v2.10.0 `KIR.dat` does NOT match the order implied by the golden.
4. **Reordering test**: reordering the fetched `kir_rna_seq.fa` records into gene-alphabetical blocks
   (stable within each gene) reproduces the golden's row order exactly and matches 13/17 rows
   byte-for-byte on the numeric fields. The remaining 4 rows (`KIR2DL2`, `KIR2DL3`, `KIR3DL2`,
   `KIR3DL3`) differ by very small amounts — score deltas in the 5th decimal place, and read-count
   deltas of ±1 on near-tied alleles (plus one row, `KIR3DL2`, gains an extra populated 3rd-candidate
   column ours-only) — consistent with reordering the flat cross-gene allele index shifting
   floating-point summation order / EM tie-breaks on already-close calls. All called top alleles
   (the actual genotype!) agree in every row; only the tie-break-sensitive secondary statistics
   differ.
5. All rows also show a **formatting difference**: our build always writes all 3 trailing
   tab-delimited columns (`fprintf(fpOutput, "\t%s", bufferAllele[j])` for `j=0..2` unconditionally),
   leaving a trailing `\t` before `\n` when the 3rd (secondary-candidate) field is empty; the shipped
   golden has no trailing tab on such rows. This is present on every row, independent of the ordering
   experiment above.

**Conclusion**: the only publicly discoverable v2.10.0 KIR reference (single commit, single archive,
in the repo upstream's own README points to) reproduces the *genotype calls* but not the *exact bytes*
of the shipped golden. This looks like a reference-file provenance gap (the exact byte-identical file
used to originally generate `T1K_example_genotype.tsv` is not recoverable from what's published) rather
than a code or wiring defect — the algorithm, thread-determinism, and top-allele calls all check out.
Per the task's explicit instruction not to fudge the test, the end-to-end byte-identity test against
the shipped golden was NOT written; see the task report for the reported status.
